# 2026-04-30 開発ログ

## pending pool seed label の取得・反映を spec §7.6 準拠に修正

### 背景
- protocol v1.0 (rev3pool) の §6.1, §7.6 は **pending プールが任意で seed label を持てる** と規定し、
  `GET /labels/{id}/download` は **submitted → fixed → pending** の優先順で検索して 404 で返すと約束している
- しかし iPad 側は `HILServerClient.listImages()` の `hasLabel` フラグ（= submitted ∪ fixed）でガードしており、
  pending の seed label を一切 fetch していなかった
- 結果、HITL の最大の利点である「サーバー側の参考 seed を編集して仕上げる」フローが死に体になっていた

### 変更ファイル
1. **`AnnottyHIL/ViewModels/HILViewModel.swift`** — pending seed の取得・反映を実装
   - `MARK: - Private Helpers` に `fetchAndSaveLabelIfAvailable(imageId:)` を新設
     - サーバーから label をダウンロードし、`ProjectFileService.saveAnnotation` で保存
     - 404 (= label がどのプールにもない) は spec §7.6 が約束する正常系なので silent skip
     - 他のエラーは log のみ（image 本体は load 済みなので user-facing error にしない）
   - `importImages` (旧 line 159-167) — `hasLabel` ガード付きの label 取得ブロックを削除し、
     ヘルパー 1 行 `await fetchAndSaveLabelIfAvailable(imageId: imageId)` に置換
   - `loadImageIntoCanvas` (旧 line 246 直後) — `canvasVM.importImage` の後に
     ヘルパー呼出 + `_ = canvasVM.navigateToImage(named: imageId)` を追加
     - `importImage` は内部で即 `loadCurrentImage()` を走らせるため、annotation 保存後に再 navigate しないと
       seed が画面に出ない。ここだけ末尾の navigate ループ保険がないので明示的に refresh

### 設計判断
- **listImages の `hasLabel` フィールドは触らない (提案 3-(c) 採用)** — フィールド名のセマンティクスは
  「submitted ∪ fixed のみ」のまま。listImages を 3 プール union に変更したり N+1 で `imageMeta` を叩いたりすると
  通信量・実装複雑度が悪化する。import 時に常に downloadLabel を試みれば pending seed は届く
- **catch パターンは `HILError.serverError(statusCode: 404, message: _)`** — labeled wildcard。
  Swift の labeled associated value pattern に正式対応した書式
- **ヘルパーは戻り値・throws なし** — image 本体は既に load 済みなので、seed 取得失敗で全体を fail
  扱いにすると "image は表示されているのに失敗" という UX 混乱を生む

### 動作確認時に注視すべき追加リスク
- **`ColorMaskParser.swift:24` が背景色を `0xFFFFFF` (白) にハードコード** している一方、
  protocol §5.1 は `(0,0,0)` (黒) を背景と定義している
- サーバーが spec 準拠の RGB PNG (黒背景) を seed として返した場合、iPad の ColorMaskParser は
  黒をクラス色として誤認識する可能性がある
- 今回の seed 取得修正で初めてこの不整合が顕在化するため、動作確認では「seed の黒背景部分が
  クラス色として塗られていないか」を必ずチェックする
- 誤認識が確認されたら、ColorMaskParser または server 出力のどちらかを spec に合わせる別タスクが必要

## protocol.md を実装の事実に合わせて clarify (rev2)

### 背景
- iPad のインポートは無事動いた → サーバーが返す seed PNG とクライアントが parse する規則は **白背景** で噛み合っている
- ところが `protocol.md` §5.1 は背景を `(0, 0, 0)` 黒と記載していた。proposal の「related concerns」で指摘された通り spec と実装が乖離していた
- iPad の実コード:
  - `ColorMaskParser.swift:24` `static let backgroundColor: UInt32 = 0xFFFFFF` (白)
  - `ColorMaskParser.swift:155-159` `isNearWhite` でアンチエイリアス対策（成分 ≥ 250 を白扱い）
  - `PNGExporter.swift:54` `pixels = [UInt8](repeating: 255, ...)` で白初期化、各クラス色だけを上書き
  - `postConfig` の callsite は **存在しない** → クライアントが palette を送らないケースもありうる前提

### 変更ファイル
1. **`docs/protocol.md`** — 実装に追従して clarification（互換破壊なし）
   - §1 用語: Class 0 (background) を `(0, 0, 0)` から `(255, 255, 255)` に
   - §5.1 マスク: 背景を白に。RGBA PNG (α 無視) も明示。ニアホワイト許容 (成分 ≥ 250) を追加。
     - クライアント実装注意: ピクセルバッファを白で初期化して palette 色を上書きすることを推奨
     - サーバー実装注意: legacy 単一チャンネル grayscale PNG (各画素値 = クラス ID) の submit を許容してよい。
       受け取ったら palette を適用して RGB PNG として保存。配信側 `/labels/{id}/download` は常に v1 仕様で返す
   - §5.2 palette: `palette[0]` の慣習を `[255, 255, 255]` に明記。`POST /config` を呼ばないクライアントの存在も明記
   - §7.2 サンプル: palette[0] を `[0,0,0]` から `[255,255,255]` に
   - §7.6: クライアント実装注意を追加 — pool でフィルタせず常に download を試み、404 を握りつぶす
   - §8 クライアント実装ガイド: 上記 §7.6 ルールを bullet 8 として追加。送信時の白初期化方針を明記
   - §9 サーバー実装ガイド: legacy grayscale PNG 受容ルールと、palette 未設定時の内部規定 palette を明記
   - §10 curl 例: palette[0] を白に
   - §11 変更履歴: rev2 (2026-04-30) のエントリを追加。互換破壊なしの追認的改訂であることを明記

### 設計判断
- **spec を実装側に寄せる**（parser を spec に直すのではない）— 既に動いている iPad の振る舞いが
  ユーザー検証済（"インポートはうまくいきました"）であり、ColorMaskParser の挙動はアンチエイリアス対応も
  含めて実用的に成熟している。spec を黒背景に固執すると ColorMaskParser・PNGExporter 双方を改修する必要があり
  影響範囲が大きい
- **互換破壊なし**: protocol_version は 1.0 のまま据え置き。サーバー実装も既存 iPad 実装も
  この clarification の前後で挙動を変える必要がない（spec が実装を追認しただけ）
