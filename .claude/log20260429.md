# 2026-04-29 開発ログ

## 標準プロトコル v1.0 策定とリポジトリ全面追従

### 背景
- 別 PC のローカルで動かしている `PeriorbitAI HIL Server`（眼周囲セグ用、7 クラス）と、
  本リポジトリの iPad アプリの間で API が食い違っており、Cloudflare 越しに接続すると
  `"The data couldn't be read because it is missing."` でデコードエラーになっていた
- 原因: サーバー側がデータセット差し替え（網膜→眼周囲）に伴い `/info`、`/images`、`/status` の
  レスポンス形式を変更していたが、iPad 側 `Codable` モデルが旧仕様のままだった
- データセットがプロジェクトごとに変わる（クラス数・色も可変）前提で、
  単一の真実源（プロトコル仕様書）を中心に iPad とサーバーを揃える方針に統一

### 変更ファイル
1. **`docs/protocol.md`**（新規）— プロトコル v1.0 仕様書
   - エンドポイント全 14 種の JSON スキーマ、curl 例、ステートマシン、移植向けチェックリスト
   - `protocol_version` フィールドで MAJOR/MINOR 互換性を明示
   - マスクは RGB PNG（カラー、palette 着色済）、Background クラスは塗らない（(0,0,0) のまま）
   - palette の所有権はクライアント側、`POST /config` でサーバーへ通知
   - エラー形式は `{"detail": "..."}` に統一、`/status` は `state` のみ必須＋進捗フィールドは optional
   - `/models/latest` は `X-Model-Version` `X-Model-Md5` `X-Model-Updated-At` ヘッダで差分同期

2. **`server/main.py`** — プロトコル v1.0 リファレンス実装に全面書き換え
   - FastAPI title を `Annotty HIL Server`、`PROTOCOL_VERSION = "1.0"` 定数追加
   - `client_config`（palette / class_names / num_classes）をモジュール状態として保持
   - `training_status` のキーを `state` に変更、`metric_name` を導入（dice 固定をやめる）
   - `validate_image_id` を JPG/JPEG にも対応、エラーは `HTTPException(detail=...)` で統一
   - `list_pool / find_pool / image_meta / model_info` ヘルパーで I/O を整理
   - `/models/versions` 系は v1 から削除（履歴管理は当面サーバー実装の裁量）
   - `/models/latest` レスポンスに `X-Model-*` ヘッダ追加

3. **`AnnottyHIL/Services/HIL/HILServerClient.swift`** — v1 準拠に全面書き換え
   - `ServerInfo / Counts / ModelInfo / ImagesEnvelope / ImageMeta / TrainingStatus` を v1 形式で再定義
   - 既存 UI（`HILDashboardView` 他）が読む `totalImages / labeledImages / unlabeledImages / modelLoaded / status / bestDice`
     は計算プロパティで互換維持
   - `getInfo()` 内部で `protocol_version` の MAJOR チェック → 不一致なら `HILError.protocolMismatch`
   - `postConfig(palette:classNames:numClasses:)` 新規追加（クライアント設定の同期 API）
   - `listImages()` は `pool=unannotated` と `pool=completed` を並列取得して合成（`ImageInfo` の `hasLabel` 判定に使用）
   - `imageMeta(imageId:)` / `infer(imageId:)` 新規 API
   - `downloadLatestModel()` の戻り値を `ModelDownload`（zip + version + md5 + updatedAt）に変更
   - エラーパースで `detail` キー優先（旧 `error` / `message` も互換のため許容）
   - `HILError.decodingError(path:underlying:)` 追加 → 「どのパスで何のキーが欠けたか」を UI に出せる

4. **`AnnottyHIL/ViewModels/HILViewModel.swift`** — `downloadLatestModel()` 戻り値変更に追従
   - `ModelDownload` から `zipData` を取り出し、`version` はサーバーヘッダ優先

### 設計判断メモ
- **palette の所有権を iPad 側にした理由**: 「色は iPad 側で設定されているものに準拠」というユーザー要件を完全に守るため。
  `/info` から `palette` を返さず、`POST /config` で iPad → server に伝える方式に統一。
  サーバーは受け取った palette を保持し、訓練・推論・マスク変換に使う。
- **マスクは RGB PNG 確定**: インデックス画像 PNG（1ch、画素値=クラスID）は技術的には軽量だが、
  Preview.app で開いたとき真っ黒に見えるのでデバッグ視認性が低い。RGB PNG なら開いて即クラス分布が分かる。
- **互換エイリアス戦略**: 既存 UI ファイル（5 ファイル）の修正を最小化するため、
  `ServerInfo` / `TrainingStatus` に旧フィールド名を計算プロパティとして残した。
  これにより `HILDashboardView` `RightPanelView` `TrainingStatusView` `ServerSettingsView` `MainView` は無修正。

### 実装上の重要点
- サーバー実装で `palette` 未設定時、`/submit` `/infer` は HTTP 503 を返す
  （理由: マスクの色 → クラスID 変換ができないため）
- `palette` 変更要求は `completed_count > 0` のとき HTTP 409 で拒否（既存マスクとの整合性破壊を防ぐ）
- `ImageInfo.hasLabel` は v1 仕様の `/images` レスポンスには直接含まれないため、
  クライアント側で `pool=completed` の `Set` と突き合わせて判定

### 残作業（別 PR 推奨）
- 多クラス対応の Canvas UI 改修（クラス選択パレット、palette 動的色塗り、インデックス画像 I/O 化）
- アプリ起動時に `postConfig` を自動呼び出しする ViewModel 側のフロー
- `HILSettings` に `palette / classNames / numClasses` の永続化と編集 UI
- `ModelSyncManager` を md5 ベースの差分判定に置き換え（現在は version 文字列）

## 同日追加: プール定義を 3 段に再編（v1.0 破壊改訂）

### 背景・要件
- ユーザー報告: **既にアノテーション済みの画像を初期画像として使うケース**で、現状の 2 プール（unannotated/completed）構造だと
  「`unannotated/labels/` に label を入れた瞬間 HITL 後扱いになる」問題が生じる
- 新要件: 画像を 3 段階に分離したい
  - **HITL 前**: label を持っていても HITL 後カウントしない、学習にも使わない
  - **HITL 後**: iPad の `Submit` で確定、再 DL/再編集可、学習に使う
  - **固定データ**: 既存の完成済データセット、HITL 不要、学習に使う

### 変更ファイル
1. **`docs/protocol.md`** — プール定義を 2 段から 3 段に
   - §2 用語、§6.1 ステートモデル（pending/submitted/fixed の 3 種、遷移ルール表）
   - §6.2 `has_seed`（pending のみ true）、`has_annotation`（submitted/fixed のいずれか）の意味再定義
   - §6.3 学習データ取得を「`submitted ∪ fixed`、pending は完全除外」と明記
   - §7.1 `/info.counts` を `{pending, submitted, fixed, total}` に
   - §7.3 `/images?pool=` のデフォルトを `pending` に、3 値に拡張
   - §7.6 `/labels/{id}/download` 検索順を `submitted → fixed → pending` に
   - §7.8 `PUT /submit/{id}` 挙動表を新規追加（pending→移動、submitted→上書き、fixed→409、不在→404）
   - §7.9 `/next` の候補を `pending` のみに限定
   - §9 サーバー実装ガイド #9 に「3 プールの物理分離」「pending → submitted の物理移動」「fixed への書き込み禁止」「pending seed の学習除外」を明記
   - §11 変更履歴に "1.0 (rev)" 行を追加（破壊改訂、PeriorbitAI 移植前のため `protocol_version` は 1.0 のまま）

2. **`server/config.py`** — パス定数を 3 プールに再編
   - `PENDING_IMAGES_DIR / PENDING_LABELS_DIR`
   - `SUBMITTED_IMAGES_DIR / SUBMITTED_LABELS_DIR`
   - `FIXED_IMAGES_DIR / FIXED_LABELS_DIR`
   - 旧 `UNANNOTATED_* / COMPLETED_*` は互換エイリアス（`scripts/` の import 互換維持のみ目的）
     - `UNANNOTATED_*` → `PENDING_*` にマップ
     - `COMPLETED_*` → `FIXED_*` にマップ（旧 retinal の `images_completed/` の意味は新 `fixed/` に最も近い）

3. **`server/data_manager.py`** — 完全書き換え
   - 例外型 `ImageNotFoundError`、`PoolReadOnlyError` 新設
   - `POOLS = ("pending", "submitted", "fixed")`、`_pool_dirs(pool)` で (images_dir, labels_dir) を返す
   - `find_pool(image_id)`: submitted → fixed → pending の順で検索
   - `get_label_path(image_id)`: 同優先順で label を検索（仕様 §7.6 準拠）
   - `submit(image_id, mask_data)`:
     - pending: 画像を pending → submitted へ `shutil.move`、pending seed があれば削除、submitted/labels に書込
     - submitted: submitted/labels のみ上書き
     - fixed: `PoolReadOnlyError` を投げる
   - `get_next_pending(strategy)`: pending プールから next sample（旧 `get_next_unlabeled` を置き換え）
   - `get_all_training_pairs()`: submitted ∪ fixed のみ。pending は **絶対に含めない**
   - `get_stats()`: `{pending, submitted, fixed, submitted_labels, fixed_labels, pending_seeds}`
   - label 拡張子は常に `.png`、画像は `.png/.jpg/.jpeg` 対応

4. **`server/main.py`** — 新 DataManager API に追従
   - `image_meta(image_id)` を DataManager 経由に
   - `/images` の pool バリデーションを `DataManager.POOLS` で
   - `/submit/{id}` を `dm.submit()` 呼び出しに簡略化、例外を HTTPException に翻訳
   - `/next` は `dm.get_next_pending()` を呼ぶ
   - `/info.counts` を `{pending, submitted, fixed, total}` に
   - `POST /config` の palette 変更禁止条件を `submitted + fixed > 0` に変更

5. **`AnnottyHIL/Services/HIL/HILServerClient.swift`** — 3 プール対応
   - `Counts` 構造体を `{pending, submitted, fixed, total}` に変更
   - `ServerInfo` の互換エイリアスを更新:
     - `labeledImages` → `counts.submitted + counts.fixed`
     - `unlabeledImages` → `counts.pending`
   - `listImagesEnvelope(pool:)` のデフォルトを `"pending"` に
   - `listImages()` を 3 プール並列取得に変更、`hasLabel = id ∈ submitted ∪ fixed`
   - `ImageInfo` のドキュメンテーションを「pool != pending を意味する hasLabel」に更新

### 設計判断メモ
- **物理移動 vs フラグ管理**: 物理移動（`shutil.move`）を採用。`ls server/data/submitted/images/` がそのまま「HITL を経た画像一覧」になる運用上の透明性を優先
- **pending seed の運命**: 提出時に削除する（submitted/labels が新規 label として上書きするため、seed は不要に）
- **PeriorbitAI 移植前のため `protocol_version` は 1.0 のまま破壊改訂**: 別 PC のサーバー実装にまだ反映していない段階での再設計なので、minor up（1.1）でなく 1.0 の中身を直接書き換える方が混乱が少ない
- **互換エイリアス（旧 `UNANNOTATED_*` 等）を残した理由**: `scripts/` の古い管理ツールが import エラーで動かなくなることを避けるため。新規コードは新定数のみ使う
- **`labeledImages` 互換エイリアスの意味**: 旧 UI は「label 完了枚数」として `info.labeledImages` を読む。新仕様では「最終確定 label を持つ画像 = submitted + fixed」が最も近い意味なので、`counts.submitted + counts.fixed` にマッピング

### 残作業
- `server/scripts/migrate_data.py` を旧 retinal 構造（`images_completed/`, `images_unannotated/`）から新 3 プール構造へのマイグレータに書き直し
- `server/scripts/import_images.py` を `pending` 直接投入に対応
- ダッシュボード UI に「pending / submitted / fixed それぞれの枚数」を表示する改修（現状は labeled/total のみ）
