# 2026-05-01 開発ログ

## class-id label の事前 upscale ガイドラインを追加 (protocol v1.0 rev3)

### 背景
- iPad は `AnnottyHIL/Models/InternalMask.swift` で **内部マスクを高解像度** (`min(2.0, 4096/maxEdge)` 倍) に持つ
- `AnnottyHIL/Services/FileManager/AnnotationLoader.swift:92-105` の `scaleMask` は **NEAREST** で server から来たマスクをアップサンプリング
- class-id データに bilinear/bicubic を使うと中間値（存在しないクラス）が生まれるため NEAREST 一択
- 結果: 1× で 1px の階段が iPad 上で 2×2 ブロックに膨らみ「ギザギザが荒すぎる」と知覚される
- これは **すべての server 実装が踏む落とし穴**で、PeriorbitAI fork で実証済み（CFD 716×297 で大幅改善）

### 変更ファイル
1. **`docs/protocol.md`**
   - §5.1 マスクのサイズ: 「入力画像と同じ」→「同じ or 整数倍にアップサンプリングしたサイズ」に拡張、§9.10 を参照
   - §9 サーバー実装ガイド: 項目 10 を新設。iPad の `calculateDimensions` 公式の引用、per-class Gaussian + argmax 手順、bilinear/bicubic を素のクラス ID 配列に直接適用してはいけない理由、低解像度 source の限界を明記。`server/scripts/smooth_labels.py` を参考実装として参照
   - §11 変更履歴に rev3 行を追加。互換破壊なしであることを強調

2. **`server/scripts/smooth_labels.py`** (新規)
   - 再利用可能な offline 一括変換ツール
   - `data/pending/labels/*.png` を iPad のターゲット解像度に変換し in-place で上書き
   - 冪等: ターゲット解像度に既に到達している label は skip
   - CLI: `--labels-dir`, `--images-dir`, `--sigma`, `--max-scale`, `--max-mask-edge`, `--limit`, `--dry-run`
   - PeriorbitAI fork で 2824 件処理 (3m18s, 0 error)

### 設計上の決定
- **server 側で事前 upscale する** vs **iPad 側で class-aware upscale する** で前者を選択
  - iPad 側修正は Swift 必要 + 複数 client への追従が必要
  - server 側 1 箇所で済むうえ、protocol 仕様の方針 (server が wire format を決める) に合致
  - 後方互換: server が 1× を返し続けても iPad は従来通り動作
- **σ=1.5 デフォルト**
  - σ=1.0 は控えめすぎ、σ=2.0 以上は細部 (caruncle 等) が縮む
  - 1.5 が視覚改善と detail 保持のバランス点
- **smoothing は class-id storage には適用するが、wire format での RGB rendering は palette 適用のみ**
  - storage = 真の class IDs (smooth-quantized at target res)
  - wire = palette 適用済 RGB at target res
  - iPad 側で再 quantize する際の精度を最大化

### iPad 側で実証された効果
- 716×297 (CFD): 「かなり良くなった」(user feedback 2026-05-01)
- 247×95 (CelebAMask-HQ crop): 限定的改善 — source 解像度の物理的限界
- 細部 (small caruncle, lid corner) は σ=1.5 で保持される

### 限界と今後
- source 解像度が極端に低い画像では 2× 化しても改善は限定的
- CelebAMask-HQ 1024×1024 ソースから再 crop すれば celeb 系もきれいに
- iPad 側 `InternalMask.maxDimension = 4096` を引き上げて 4× 化等もありえるが、メモリ消費 16 倍
